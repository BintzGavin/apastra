# Black Hole Architecture

The PromptOps system adopts the Black Hole Architecture, ensuring a highly auditable, file-based foundation for managing prompt assets and their execution lifecycles.

## The Core Principle

The defining characteristic of a Black Hole Architecture is that **file-based durable state is the absolute source of truth**.

Computation is considered stateless and entirely replaceable. Systems and agents read from explicit, declarative files, perform work, and generate append-only, immutable derived files. They do not manage or rely on hidden, internal state mechanisms (e.g., undocumented SaaS databases).

## Architectural Mapping

### Durable State (Source of Truth)
All foundational definitions reside in Git as versioned files. This includes prompt specifications, datasets, evaluators, suites, harness adapters, regression policies, delivery targets, and consumption manifests.

### Stateless Compute (Workers)
The execution layer relies on workers—GitHub Actions jobs, self-hosted runners, internal schedulers, notebooks, or CLIs. These workers read `run_request` structures and emit `run_artifact` directories. Crucially, they do not own hidden state and can be swapped or scaled horizontally without impacting the underlying architectural truth.

### Append-Friendly Immutable Artifacts (Derived State)
Outputs generated by stateless workers must be treated as immutable records. This includes run artifacts, regression reports, and promotion records. To prevent Git repository bloat, small metadata indices are stored in Git (often on an `artifacts` branch), while large raw outputs (like transcripts or traces) are referenced by content digest and stored in an open-ended artifact backend (e.g., OCI registry or object storage).

### Human Checkpoints
Clear boundaries are defined for human intervention and approval. PR reviews govern change, and explicit approvals are required for promotions or policy alterations using branch protection rules and CODEOWNERS files.

### Recovery and Replay
Because the entire system relies on reproducible, content-addressed files, any historical evaluation run can be replayed by resolving the identical digested inputs and harness version, accounting only for the natural variance of the underlying models.
