#### 1. Context & Goal
- **Objective**: Include provenance data in the metadata returned by the minimal runtime `resolve()` function.
- **Trigger**: The vision document explicitly requires provenance metadata to be part of the system's core output.
- **Impact**: Enables downstream consumers like EVALUATION harnesses to record and verify provenance for benchmark runs and run artifacts.

#### 2. File Inventory
- **Create**: None
- **Modify**: `promptops/runtime/resolve.py` (Update the `resolve()` function to extract and return provenance data in the metadata dictionary)
- **Read-Only**: `docs/vision.md`, `promptops/schemas/run-manifest.schema.json`

#### 3. Implementation Spec
- **Resolver Architecture**: The `resolve()` function will extract `provenance` from the resolved prompt spec or the manifest (if applicable) and include it in the returned `metadata` dictionary.
- **Manifest Format**: Unchanged.
- **Pseudo-Code**:
  - After resolving the prompt spec via `ResolverChain().resolve()`, check if `provenance` exists in the resolved spec dictionary or in the manifest rules for the given prompt ID.
  - If `provenance` data is found, add it to the `metadata` dictionary before returning `rendered_prompt, metadata`.
- **Harness Contract Interface**: N/A
- **Dependencies**: `promptops/schemas/run-manifest.schema.json`

#### 4. Test Plan
- **Verification**: Run `python -c "from promptops.runtime.resolve import resolve; res, meta = resolve('test', 'local:./prompts/'); print('provenance' in meta)"` with a mock prompt containing provenance data.
- **Success Criteria**: The output prints `True` and the metadata dictionary includes the expected provenance data.
- **Edge Cases**: Prompts without provenance data should not raise errors and should simply omit the `provenance` key in the metadata output.
